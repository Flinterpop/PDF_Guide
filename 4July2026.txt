     Per-PDF Named Bookmarks for PDF Sherpa

     Context

     PDF Sherpa remembers your last-read page per PDF, but there's no way to
     mark and name multiple places in a document. This adds user-created named 
     bookmarks, per PDF: bookmark the current page (default name "Page N",
     renameable), see them at the top of the Topics pane, click to jump,
     right-click to rename/delete. Bookmarks are stored in a sidecar file next 
     to each PDF (manual.pdf → manual.bookmarks.json) so they travel with the
     folder — consistent with the app's existing .toc/.json topics-sidecar
     convention.

     All code changes are in the single-file app C:\source_py\PDF_Guide\app.py
     (class PDFSherpaApp), plus docs/version files.

     Data model

     - Sidecar: os.path.splitext(pdf_path)[0] + ".bookmarks.json". The double
     extension can't collide with find_metadata_path (app.py:108), which only
     checks base + ".toc" / base + ".json".
     - Format — a bare JSON list, mirroring the topics .json format
     (app.py:147), 1-based pages, hand-editable:
     [{"page": 12, "title": "Wiring diagram"}]
     - In memory: self._bookmarks: list[dict] kept sorted ascending by page.
     Duplicate pages allowed (labels differ).

     Changes

     1. Module-level helpers (after load_metadata, ~app.py:165)

     - bookmarks_path(pdf_path) -> str — path scheme above.
     - load_bookmarks(pdf_path) -> list[dict] — missing file → [] silently;
     validate entries (int(page) >= 1, str(title)), sort by page;
     corrupt/unreadable → messagebox.showwarning + [] (next save overwrites —
     same spirit as save_config's never-crash-on-write).
     - save_bookmarks(pdf_path, bookmarks) -> bool — json.dump(..., indent=2),
     UTF-8. Empty list → delete the sidecar if present (no file = no bookmarks,
     keeps folders clean). OSError → warning "kept for this session only",
     return False, keep in-memory state.

     Add simpledialog to the tkinter import at app.py:45.

     2. State (__init__, near self._topic_entries ~app.py:189)

     self._bookmarks = [] and self._bm_menu_index = None (row targeted by
     right-click).

     3. Load + Topics-tree integration

     - load_topics (app.py:944): first line, self._bookmarks = 
     load_bookmarks(pdf_path). This is the single choke point — every PDF switch
     goes through on_pdf_selected → load_topics, and all its branches end in
     _populate_topic_tree().
     - _populate_topic_tree (app.py:971): restructure so bookmarks render before
     the placeholder/topics logic:
       a. If self._bookmarks (after applying the existing filter needle to
     bookmark titles): insert parent iid="bm-root", text "Bookmarks",
     values=("",), open=True; children iid=f"bm-{i}" (index into
     self._bookmarks), text=title, values=(page,).
       b. Then the placeholder branch — no longer an early return before
     bookmarks, so a PDF with no .toc still shows its bookmarks above "(no
     metadata file found)".
       c. Topics loop unchanged (Tk auto-iids I001… can't collide with bm-*).
       d. "(no matches)" row only when the needle filters out both bookmarks and
     topics.
     - on_topic_selected (app.py:1321): no functional change — bookmark rows
     carry a 1-based page in values[0] exactly like topic rows; bm-root has
     empty values and is already ignored by the existing guard.

     4. Add bookmark — add_bookmark() method (near _on_topic_search_changed 
     ~app.py:995)

     - Guard: no self.doc/self.current_pdf_path → return.
     - page = self.current_page + 1; simpledialog.askstring("Add bookmark", ...,
     initialvalue=f"Page {page}", parent=self); cancel → abort, blank →
     default.
     - Insert, re-sort by page, save_bookmarks(...), _populate_topic_tree(),
     select + see() the new row.

     Three entry points:
     - Nav-bar button "🔖" after self.save_btn (app.py:403–408), with
     _add_tooltip mentioning Ctrl+B and right-click rename/delete. Always
     enabled; no-ops without a doc (like Prev/Next).
     - Ctrl+B in _bind_shortcuts (~app.py:511): top.bind("<Control-b>", 
     self._viewer_key(self.add_bookmark)). Must use _viewer_key (app.py:535),
     not the bare Ctrl+S lambda pattern — Tk entries have a default emacs
     Control-b binding, and this suppresses the shortcut while typing in a
     search box.
     - Canvas right-click: add_separator() + "Bookmark this page" appended to
     self.canvas_menu (app.py:444–448). Safe: _on_canvas_right_click only
     entryconfigs indices 0–1.

     5. Rename / delete — Topics-tree context menu (in _build_ui after 
     topic_tree bindings ~app.py:351)

     - self.bookmark_menu = tk.Menu(self, tearoff=0) with Rename / Delete; bind
     <Button-3> on topic_tree → _on_topic_right_click.
     - _on_topic_right_click: identify_row(event.y); only for bm-<i> iids (not
     bm-root, not topics): store index, select the row, tk_popup.
     - _rename_bookmark: askstring prefilled with current title; update, save,
     repopulate, reselect.
     - _delete_bookmark: remove, save, repopulate. No confirmation (one click to
     re-add).

     6. (ann) copies — save_annotations (app.py:1279–1287)

     Alongside the existing topics-file copy, also shutil.copyfile the
     .bookmarks.json sidecar to base + "(ann).bookmarks.json" if present, same
     try/except-OSError-pass pattern.

     7. Edge cases

     - Page out of range (PDF shrank / hand-edited): keep the bookmark;
     goto_page (app.py:1377) already clamps.
     - Read-only folder: warning from save_bookmarks, bookmarks stay usable
     in-memory for the session.
     - Filter interplay: free — the tree rebuilds from self._bookmarks on every
     filter keystroke.

     8. Docs & version

     - APP_VERSION = "1.3.0" (app.py:63) and installer.iss AppVersion in sync
     (the comment at app.py:62 requires it).
     - Module docstring feature list (app.py:1–33): add a bookmarks line.

     Ctrl+B row in the shortcuts table.
     - README.md: one feature bullet, matching the 1.2.x style (commit d785ec6).

❯ what is sash position                                                         

● The sash is the draggable divider between the Bookmarks list and the Topics
  list in the middle pane — the thin horizontal bar you can grab with the mouse
  to resize the two lists. "Sash" is just Tk's (the GUI toolkit's) name for the
  drag handle in a split pane; the same thing exists between the three main
  panes (PDFs / Topics / Viewer), vertically.

  The sash position is where that divider currently sits, in pixels from the
  top. When a PDF's bookmarks first appear, the app places it at 25% of the pane
  height. If you drag it — say you pull it down to give bookmarks half the
  space — the app remembers that pixel position and puts the divider back there
  whenever the Bookmarks list reappears (after switching PDFs, or after deleting
  all bookmarks and adding one again).

  What I meant by "session-only" in the last message: that remembered position
  lives in memory, not in the config file, so when you close and relaunch the
  app the divider resets to the 25% default. If that bothers you, it's a
  two-line change to save it in %APPDATA%\PDFGuide\config.json alongside the
  window geometry.

❯ is there a problem with long lines in the markdown                            
● Let me check how the app's custom Help renderer actually treats line breaks
  before answering:

  Searched for 1 pattern, read 1 file (ctrl+o to expand)

● No — long lines are completely fine, and for this app they're actually the
  slightly safer format. Concretely:

  In the app's Help window: the custom renderer (_render_markdown, app.py:1909)
  does "lazy wrapping" — for paragraphs and bullets it gathers all following
  non-blank lines and joins them with spaces into one string
  (gather_continuation), and the Text widget then word-wraps to the window
  width. So a paragraph written as one 300-character line and the same paragraph
  hard-wrapped at 78 columns render pixel-identically. The screenshot I took a
  moment ago was of your rewrapped file, rendering cleanly.

  On GitHub: standard markdown treats consecutive lines as one paragraph anyway,
  so both styles display the same there too.

  The one real hazard actually favors long lines: gather_continuation stops
  joining when a line looks like the start of a new block — a line beginning
  with -, 1., #, |, or a code fence. With hard wrapping, if a break ever
  happened to land so a continuation line starts with something like -  or 2. ,
  the renderer would misread it as a new bullet mid-paragraph. One long line per
  paragraph makes that impossible.

  Two things to keep in mind with the long-line style, neither serious:

  - Table rows must stay on one line each (the renderer detects tables by | per
  line) — but that's true in any markdown, and your shortcuts table is fine.
  - Git diffs get coarser — changing one word marks the whole paragraph line as
  changed, so diffs are a bit noisier to review. That's the only cost, and it's
  cosmetic.

  So keep the rewrapped file as is — nothing needs changing.
